# 구현 · 사용자 확인 · 테스트 게이트 — 필수 워크플로

> **목적**: `plan.md`만 보고 곧바로 코딩하거나, 사용자 확인 없이 단계를 건너뛰는 일을 방지한다.  
> **적용 대상**: Cursor AI 및 모든 기여자.  
> **연계 문서**: `.cursorrules`, [`docs/rules/project_context.md`](project_context.md) §작업 프로토콜, `docs/CURRENT_WORK_SESSION.md`, Cursor 프로젝트 스킬 `.cursor/skills/idr-session-workflow/SKILL.md` (본 문서의 에이전트용 요약; 상충 시 **본 문서 우선**).

`docs/CURRENT_WORK_SESSION.md`의 **세션 워크플로 표**와 아래 **Gate A~E**는 동일한 5단계다.

**한 줄 요약**: `plan`/참조 문서 확인 → **CURRENT에 구현 상세 계획** → 사용자 승인 → **구현**·CURRENT 기록 → 사용자 승인 → **CURRENT에 테스트 상세 계획** → 사용자 승인 → **테스트 검증**·CURRENT 기록 → **WORK_HISTORY + `plan.md` + 다음 세션 CURRENT**.

---

## 요약 다이어그램

```text
plan.md·SDD·규칙 확인
    → Gate A: CURRENT에 「구현 상세 계획」작성 → ⏸ 사용자 승인 → (이후에만 코딩)
    → Gate B: 구현 + 「구현 완료 요약」·체크리스트 [x] → ⏸ 사용자 승인
    → Gate C: CURRENT에 「테스트 계획」(상세) → ⏸ 사용자 승인 → (이후에만 테스트 실행)
    → Gate D: 테스트 작성·실행·「테스트 검증 결과」기록
    → Gate E: WORK_HISTORY + plan.md 체크 + CURRENT를 다음 세션으로 교체
```

---

## 규칙 (준수 순서) — Gate A ~ E

### Gate A — 구현 상세 계획 (코딩 **전**)

1. **`docs/plans/plan.md`**, SDD, `docs/rules/*.md` 등 **참조 문서를 먼저 확인**한다.
2. `docs/CURRENT_WORK_SESSION.md`에 **「구현 상세 계획」** 섹션을 채운다.
   - 이번 세션에서 다룰 범위, 구현 순서, 파일·모듈 단위, 주의사항·의존 관계 등(`plan.md` 체크리스트보다 **한 단계 구체적**이어도 된다).
   - **기록 위치 (금지)**: 위 상세는 **`CURRENT`에만** 둔다. **`docs/plans/<새파일>.md`를 세션 작업용 상세 계획으로 신설하지 않는다.** `docs/plans/`에는 WBS(`plan.md`), 경로·아키텍처 정본(예: `lis_public_url_path_map.md`), 강의·패키지 등 **참조·백그라운드** 문서만 둔다.
3. **진행 상태**를 `구현 상세 계획 완료 — 사용자 확인 대기` 등으로 표시한다.
4. **사용자 명시 승인 전에는 코드 작성·수정을 시작하지 않는다.** (문서·코드 읽기·조사만은 가능)

### Gate B — 구현 완료

1. 승인된 상세 계획에 따라 구현한다.
2. **「구현 완료 요약」**을 CURRENT에 작성한다(변경 파일, 설계 결정, 제한 사항).
3. 해당 Phase 체크리스트 `- [x]` 반영.
4. **진행 상태**를 `구현 완료 — 사용자 확인 대기`로 둔다.
5. **사용자 검토·승인 전에는 Gate C(테스트 상세 계획)로 넘어가지 않는다.**

### Gate C — 테스트 상세 계획 (실행 **전**)

1. 사용자가 구현을 확인한 뒤, CURRENT **「테스트 계획」**에 **상세**를 적는다.
   - 범위(단위/통합/수동), 케이스·우선순위, 명령(`PYTHONPATH`, `pytest` 경로 등).
   - **Gate A와 동일**: 테스트 상세도 **`docs/plans/` 신설 파일이 아닌 `CURRENT`만** 사용한다.
2. **진행 상태**를 `테스트 계획 수립 완료 — 실행 승인 대기`로 둘 수 있다.
3. **사용자 승인 전에는 테스트 코드 작성·실행(Gate D)에 들어가지 않는다.**

### Gate D — 테스트 검증

1. 승인된 계획대로 테스트를 작성·실행한다.
2. 결과를 **「테스트 검증 결과」**에 기록한다(일시, 통과/실패, 조치).
3. 실패 시 수정 후 재실행·갱신한다.
4. 사용자 최종 확인 후 Gate E로 진행한다.

### Gate E — 이력 이전·문서 전환

1. `docs/history/WORK_HISTORY.md`에 완료 내역을 이전한다.
2. **`docs/plans/plan.md`**: 완료된 WBS `- [x]` 및 상단 「진행 현황」표 갱신.
3. `docs/CURRENT_WORK_SESSION.md`를 **다음 세션(다음 Phase) 과제만** 담도록 교체한다.

---

## 승인 범위 해석 (중요 — 과대해석 금지)

- 사용자의 「진행해라」「구현해」「OK」「승인」 등은 **문맥상 직전에 논의 중이던 Gate에만** 적용한다.
- **Gate A에 대한 승인** = *구현 착수만* 허용. 구현이 끝나면 **여기서 한 턴을 종료**하고, `CURRENT`를 **구현 완료 — 사용자 확인 대기**(Gate B)로 둔 뒤 사용자 검토를 기다린다.
- **Gate B 이후**(테스트 계획·실행)를 같은 승인으로 진행하려면 사용자가 **명시**해야 한다. 예: 「테스트까지 진행」「`make test` 돌려」「C/D 포함」.
- 애매하면 **구현만 하고 멈춘 뒤** “다음은 Gate C(테스트 계획) 승인이 필요합니다”라고 묻는다.

---

## AI(Cursor) 동작 금지 사항

- **Gate A 사용자 승인 없이** 구현(코드 변경)을 시작하기.
- **Gate A 승인만** 받고 **Gate B 완료 후 멈추지 않고** 곧바로 Gate C/D(테스트 계획·실행·검증 기록)로 넘어가기. *(구현 직후 일괄 테스트는 금지 — 별도 C 승인 필요)*
- **Gate B 사용자 승인 없이** 테스트 상세 계획(Gate C)을 작성하거나 테스트를 실행하기.
- **Gate C 사용자 승인 없이** 테스트 코드 작성·실행하기.
- 사용자 확인 없이 **Session N+1 문서로 `CURRENT_WORK_SESSION.md` 전체 교체**하기.
- **Gate D(검증 결과 기록) 없이** `WORK_HISTORY`만 갱신하고 세션을 넘기기.
- **Gate E에서 `plan.md` 체크박스·진행 표 갱신 누락**하기.
- 세션 작업의 **구현 상세 계획(Gate A) 또는 테스트 상세 계획(Gate C)** 을 **`docs/plans/`에 새 마크다운 파일로 작성**하기. *(정본: `CURRENT_WORK_SESSION.md`만.)*

## `CURRENT_WORK_SESSION.md` 표준 섹션 (권장 순서)

1. 세션 메타(Phase, `plan.md` 참조, **진행 상태**, 완료 기준)
2. 환경·전제
3. **세션 워크플로 상태** (표: Gate A~E)
4. (선택) Phase WBS 체크리스트 — `plan.md`와 동기화된 항목
5. **구현 상세 계획** (Gate A — 코딩 전에 채움; **`docs/plans/` 신설로 대체 금지**)
6. **구현 완료 요약** (Gate B)
7. **테스트 계획** (Gate C — 상세; **`docs/plans/` 신설 금지**)
8. **테스트 검증 결과** (Gate D)
9. 세션 마감(Gate E)·다음 세션 예고

세션 초기에는 5~8을 비우거나 `— 진행 시 작성 —`으로 둔다.

### 진행 상태 예시 (문서·표와 함께 쓸 값)

- `구현 상세 계획 작성 전`
- `구현 상세 계획 완료 — 사용자 확인 대기` (Gate A)
- `구현 완료 — 사용자 확인 대기` (Gate B)
- `테스트 계획 수립 완료 — 실행 승인 대기` (Gate C)
- `테스트 검증 완료 — 세션 마감 대기` (Gate D 이후, E 전)

## `project_context.md` 작업 단계와의 대응

| 프로토콜 | 대응 Gate |
|----------|-----------|
| Step 1: `plan`/참조 확인 후 구현 상세 계획을 CURRENT에 기록·승인 대기 | **A** |
| Step 2: 구현 후 구현 완료 요약·승인 대기 | **B** |
| Step 3a: 테스트 상세 계획 CURRENT·승인 대기 | **C** |
| Step 3b: 테스트 실행·검증 결과 기록 | **D** |
| 이력 이전·`plan.md`·다음 CURRENT | **E** |
